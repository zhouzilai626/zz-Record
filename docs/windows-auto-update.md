# Windows 自动更新说明

ZZ Record 使用 GitHub Release 提供 Windows 自动更新。已安装正式版的用户不需要先卸载旧版本：应用会检查官方发布页，在发现较新版本后提示下载；下载完成后由用户选择是否重启安装。

## 用户如何更新

1. 保持网络可访问 GitHub。
2. 启动 ZZ Record。应用会在后台检查更新，并在之后每 6 小时最多检查一次。
3. 在 Windows 编辑页面的左侧打开 **设置**，找到“应用更新”卡片。这里会显示当前安装版本和实际更新状态。
4. 点击 **检查更新**。该按钮会连接固定的官方 GitHub Release，不会从用户输入或网页链接读取更新地址。
5. 发现更新时，选择 **下载更新**。下载期间可以继续使用应用。
6. 下载完成后，选择 **安装并重启**。应用会退出，Windows 安装程序会替换程序文件并重新启动应用。
7. 如果正在录制，应用会拒绝重启安装，并提示先结束录制。下载好的更新会保留，录制结束后可再次选择安装。
8. 全新安装 Windows 版本时，安装程序会默认创建桌面快捷方式和开始菜单中的 **ZZ Record** 入口；如果电脑的组织策略禁止桌面图标，仍可能只显示开始菜单入口。

> 开发版会明确显示“开发版”提示，不能检查或安装正式更新；请使用 GitHub 下载并安装的正式 Windows 版本进行更新。

若桌面没有图标，可先从开始菜单搜索 **ZZ Record**。不要删除安装目录；确认问题后应使用最新安装包重新安装或在开始菜单中右键创建快捷方式。

## Windows 桌面快捷方式

Windows NSIS 安装包已明确配置为始终创建桌面快捷方式，同时创建开始菜单中的 **ZZ Record** 入口。快捷方式目标指向安装目录中的 `ZZ Record.exe`，更新时会继续沿用该安装入口。

如果安装完成后桌面仍没有图标，请先在开始菜单搜索 **ZZ Record**；若开始菜单有入口，通常是当前 Windows 用户策略或桌面图标显示设置导致。可右键该入口选择“发送到 → 桌面（创建快捷方式）”。

若开始菜单和桌面都没有入口，应保留安装包与安装日志，在干净 Windows 用户环境重新安装最新安装包并记录结果；不要把“没有桌面图标”直接判断为程序无法运行。

## 数据安全边界

更新程序只替换安装目录中的应用文件。它不会主动删除或覆盖：

- 项目文件；
- 录制视频和音频；
- 录制目录配置；
- 麦克风、系统音频与导出偏好；
- 手机摄像头配对状态。

下载或网络检查失败时，已安装版本仍可继续使用。用户可以稍后再次检查或下载。

## 每次发布必须满足的条件

GitHub Release 必须使用新的、未发布过的语义化版本标签，例如 `v1.4.9`。版本号必须与 `package.json` 一致。

每个 Windows x64 Release 必须上传下面四个同一构建产物：

- `zz-Record-windows-x64.exe`
- `zz-Record-windows-x64.exe.blockmap`
- `latest.yml`
- `SHA256SUMS.txt`

`latest.yml` 内的版本、安装包文件名、SHA-512 信息必须与实际上传的安装包一致。不要手工修改或重命名这些文件；旧客户端依赖它们寻找和验证更新。

## 发布者检查清单

1. 提升 `package.json` 与 `package-lock.json` 的版本号。
2. 执行类型检查、相关测试与完整测试；构建 Windows 安装包。
3. 核对安装包签名（若已配置签名证书）、`latest.yml`、blockmap 与 `SHA256SUMS.txt`。
4. 在 GitHub 创建新 tag 和 Release，并上传四个发布资产。
5. 从仍安装旧版本的独立 Windows 环境，在编辑页的“应用更新”卡片中检查更新、下载、重启安装。
6. 确认升级后版本号正确，项目、录制文件和设置仍在。

## 首次真实升级验收（v1.4.8 → 后续正式版本）

首次验证必须使用一台未参与构建的 Windows 电脑，并保留至少一个项目、录制文件和自定义设置。当前已完成 `v1.4.6 → v1.4.7` 的同机升级验收；仍建议以未参与构建的 Windows 电脑完成 `v1.4.8 → 后续正式版本` 的独立验收。

1. 安装当前正式版 `v1.4.8`，确认可启动，并记下一个项目文件、录制目录和一项音频或导出设置。
2. 发布一个**高于 `v1.4.8` 的正式版本**，且 GitHub Release 已公开上传四个必需附件。
3. 启动旧版应用，打开编辑页的“应用更新”卡片，选择 **检查更新**，再选择 **下载更新**；确认下载过程中应用仍可使用。
4. 下载完成后，选择 **安装并重启**；若正在录制，确认应用显示“请先结束录制”，且不会退出。
5. 结束录制后再次选择安装，确认应用重启后的版本号与 Release 标签一致。
6. 打开先前项目，检查录制文件、录制目录、音频设置和导出设置仍然存在。
7. 断网后重复“检查更新”或下载操作，确认旧版仍可启动和使用；恢复网络后可以重试。

> **注意：** GitHub 的预发布版本不是对稳定版自动更新链路的有效验收目标。稳定版客户端通常只跟随正式发布通道。要验证真实自动升级，必须使用一个公开的、较新的正式 Release；或者另行建立专用测试更新通道，并在旧版中显式启用该通道。

## 已知限制

- GitHub 在部分网络环境可能不可访问或下载较慢；这种情况下应用会保留当前版本，并允许之后重试。
- 真正的升级过程依赖 Windows 安装程序和实际旧版环境，不能由单元测试完全替代。每个新的发布通道或安装器配置变更后，必须执行一次真实升级验收。
- 自动更新仅信任 electron-builder 配置中的固定 GitHub 仓库；不要用用户输入或网页内容覆盖更新地址。
- 开发环境不会执行真实更新检查；编辑页会显示对应提示，避免误导为已经连接正式发布服务。
